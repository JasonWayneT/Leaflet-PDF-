# Decision Log — Leaflet PDF

Canonical record of decisions made during the product brief session. Every meaningful choice is logged here with context.

---

## 2026-06-18 — Product Brief Session

### DEC-001: Start from scratch, not a port of v1
**Decision:** Leaflet PDF is a new project, not an extension of v1. The v1 project folder contains documentation only (zero implementation). Some documentation may be reused (see addendum), but the codebase starts fresh.
**Reason:** V1 was over-engineered and did not go through proper idea validation. V2 starts with BMAD to validate the idea and define scope before building.

### DEC-002: Standard methodology mode (not lightweight)
**Decision:** Using the standard project methodology template.
**Reason:** Leaflet PDF has real technical complexity — an AI transformation pipeline, source fidelity validation, a custom design system, and a multi-stage output engine. Lightweight mode is appropriate for quick demos and personal tools. Standard mode is warranted here.

### DEC-003: Option B — AI restructures content, not just styles it
**Decision:** Leaflet PDF applies the Learning Document Playbook pedagogical framework to restructure content before rendering. It does not simply apply a visual template to existing structure.
**Reason:** The core value is making content more learnable, not just more attractive. Restructuring for comprehension (BLUF, mental buckets, teach-not-label headings, etc.) is the thing that earns the reading experience.

### DEC-004: Source fidelity is a hard constraint
**Decision:** Leaflet PDF must validate that all factual claims in the output are consistent with the source material. This is a non-negotiable product constraint, not a nice-to-have.
**Reason:** The AI restructuring creates hallucination risk. Leaflet PDF's trustworthiness depends on preserving the author's facts exactly. The distinction between "reorganization" and "rewriting" must be enforced technically, not just as a principle.

### DEC-005: Orbital Light and Orbital Night as v2 MVP visual styles
**Decision:** Two output styles ship in v2 MVP: Orbital Light (light ground) and Orbital Night (dark ground). Both use the same Industrial Cyber-Brutalism structural language — Anton typography, corner brackets, thick borders, JetBrains Mono labels — in different colorways. User selects style at submission time; Orbital Light is the default.
**Reason:** Both styles share the same HTML template structure and differ only in CSS variables. Adding both in v2 costs little architectural effort and gives meaningful variety. Strategyzer and other styles remain Phase 3 roadmap items.
**Supersedes:** Original DEC-005 (single style only). Corrected in architecture session 2026-06-18.

### DEC-006: Style naming — Orbital Light (light ground) and Orbital Night (dark ground)
**Decision:** Orbital Light = white/off-white base, dark text, blue accent. Orbital Night = Terminal Black (#131313) base, Acid Yellow (#c7f300) and white accents. The spec file previously named ORBITAL-LIGHT.md (which described the dark design) has been renamed to ORBITAL-NIGHT.md. A new ORBITAL-LIGHT.md will define the light-ground colorway.
**Reason:** Original DEC-006 incorrectly described Orbital Light as light-ground while the actual spec file was dark. Architecture session confirmed the dark design is Orbital Night; Orbital Light is the white-base version. Naming corrected to match design intent and spec files.
**Supersedes:** Original DEC-006 (contained incorrect description of ORBITAL-LIGHT.md). Corrected in architecture session 2026-06-18.

### DEC-007: YouTube URL input included in MVP
**Decision:** YouTube URL is an MVP input type alongside copy/paste, .md, and .txt files.
**Reason:** YouTube transcripts are a primary content format Jason wants to transform (explainer videos, class recordings, etc.). Technically feasible via `youtube-transcript-api` without requiring Gemini or complex scraping.

### DEC-008: Critical thinking layer deferred to Phase 4
**Decision:** The critical thinking prompts feature (Brown & Keeley framework) is roadmap, not MVP.
**Reason:** It is a different mode of interaction from the core transformation loop. Adding it to v2 would blur the product's identity before the core value is validated. Lower hallucination risk than the alternatives callout feature (questions, not assertions), so it is the safer of the two roadmap features to build next.

### DEC-009: Alternatives/context callout deferred — hallucination risk
**Decision:** The feature that surfaces alternatives the author didn't mention (e.g., "this article recommends X but also consider Y") is not on the near-term roadmap until source fidelity is proven.
**Reason:** This feature requires the AI to generate knowledge beyond the source material, which directly conflicts with the source fidelity constraint in DEC-004. Do not build until the core fidelity architecture is validated and a safe approach is designed.

## 2026-06-18 — PRD Session

### DEC-010: Desktop app for MVP
**Decision:** Leaflet PDF is a Windows desktop application — self-contained, no server required, distributed on GitHub.
**Reason:** Primary user is a single person (Jason). Desktop app is the right form factor for a personal utility tool that processes files and produces local PDF output.

### DEC-011: CLI deferred to roadmap
**Decision:** Command-line interface is not in scope for v2 MVP.
**Reason:** Intended for Hermes agent integration (Phase 5). The desktop app is the MVP surface; CLI can be added as a wrapper once the core pipeline is stable.

### DEC-012: Windows is the target OS for MVP
**Decision:** v2 targets Windows 11. No Mac or cross-platform support in MVP.
**Reason:** Primary user is on Windows 11. GitHub distribution for others is a secondary goal; cross-platform support can be added later if demand exists.

### DEC-013: Automatic retry on fidelity failure — 2 retries (3 total attempts)
**Decision:** When Source Fidelity Validation fails, the pipeline automatically retries Transformation up to 2 times before halting with an error. Total maximum attempts: 3.
**Reason:** A single fidelity failure may be a prompt quality issue that resolves on retry. But unlimited retries would burn tokens and block the user indefinitely. 3 attempts balances reliability with cost and UX.

### DEC-014: Deterministic technique selection, AI execution only
**Decision:** Which Learning Design Framework techniques to apply is determined by rules (deterministic), not AI judgment. The AI is only responsible for executing the selected techniques.
**Reason:** Keeps token usage bounded, output structure predictable, and decision logic auditable. Reduces cost by preventing AI from re-evaluating technique selection on every run.

### DEC-015: Technique visibility — accessible but not prominent
**Decision:** Users can see which techniques were applied to a document via a secondary action (info panel, details view). This information is not displayed in the main output view.
**Reason:** Most users won't care most of the time. Making it discoverable respects the users who do care without cluttering the primary experience.

### DEC-016: Named spec files as canonical Visual Style sources
**Decision:** Each Visual Style is fully specified in its own named file: `ORBITAL-LIGHT.md` (light ground) and `ORBITAL-NIGHT.md` (dark ground, formerly `ORBITAL-LIGHT.md`). The PRD references these files rather than duplicating style rules inline.
**Reason:** Style details evolve during implementation. Single authoritative files per style prevent the PRD and specs from diverging. Each future Visual Style gets its own named spec file. File renamed in architecture session 2026-06-18.

## 2026-06-18 — Architecture Session

### DEC-018: Tech stack — Electron + TypeScript
**Decision:** Leaflet PDF is built as an Electron + TypeScript desktop app, scaffolded with electron-forge (Vite + TypeScript template), React in the renderer process, Playwright for PDF generation.
**Reason:** UI flexibility for frontend design; Playwright renders HTML/CSS to PDF which maps directly to the Visual Style spec format; `youtube-transcript` npm package is a JS-native YouTube extraction solution requiring no Python subprocess; `@anthropic-ai/sdk` TypeScript SDK is first-class.

### DEC-019: BYOA multi-provider AI with two model slots
**Decision:** Users supply their own API keys. Two model slots: Transformation (quality-critical, default `claude-sonnet-4-6`) and Validation/Utility (speed/cost, default `claude-haiku-4-5`). Providers in v2: Anthropic, Google Gemini, Ollama (local). SDK: Vercel AI SDK (`ai` package) for provider-agnostic interface. API keys stored in OS keychain via `keytar`; non-sensitive config in `electron-store`.
**Reason:** Keeps running costs on the user, supports local/offline via Ollama, and avoids Leaflet PDF needing to manage API infrastructure. Two model slots balance quality on the critical transformation call with cost on mechanical tasks.

### DEC-020: Monorepo structure with `core` package for MCP readiness
**Decision:** npm workspaces monorepo from day one: `packages/core` (all pipeline logic, zero Electron dependencies), `packages/electron-app` (portfolio app, imports from core), `packages/mcp-server` (Phase 2 roadmap). Orchestrator emits events via Node.js built-in `EventEmitter`; Electron shell bridges to IPC; MCP shell bridges differently. `core` never imports from Electron.
**Reason:** Jason's primary day-to-day use case is Leaflet PDF as an MCP server (via Cursor, Claude Desktop, or any MCP-compatible host). The Electron app is the GitHub portfolio piece. Monorepo makes MCP conversion a wrapper, not a rewrite. EventEmitter abstraction is correct architecture regardless of MCP plans.

### DEC-021: Pipeline module contracts and orchestrator ownership of retry logic
**Decision:** Six modules: Intake, Technique Selector, Claim Extractor, Transformer, Validator, Renderer — all stateless with defined input/output contracts. Pipeline Orchestrator owns sequence and retry logic (max 3 Transformation+Validation attempts). Every module emits `{ stage, cause, retryable }` on failure. Orchestrator fires typed IPC events for UI progress updates.
**Reason:** Stateless modules with explicit contracts enforce FR-6 (named-stage errors) and FR-21 (style extensibility). Retry logic in the orchestrator keeps modules simple and testable independently.

### DEC-022: AI-based Factual Claim extraction (Validation model slot)
**Decision:** Claim extraction uses a dedicated AI call on the Validation/Utility model slot. Prompt instructs semantic-level extraction (meaning, not literal strings) so Validator checks intent preservation, not exact phrasing.
**Reason:** Highest accuracy for the non-fiction content types Leaflet PDF handles. Token cost at Haiku/Flash pricing is negligible. Rule-based NLP risks brittle extraction that produces false validation failures — wrong trade for a tool where fidelity is the core trust signal.

## 2026-06-19 — MCP Sampling Session

### DEC-023: MCP Sampling as a fourth AI provider mode
**Decision:** Add `'mcp-sampling'` as a fourth `ProviderConfig` variant in `@leafletpdf/core`. When active, the MCP server delegates all AI inference to the MCP host via the `sampling/createMessage` protocol request, rather than calling an external AI API directly. Auto-detected when no API key and no Ollama URL are configured. Explicit opt-in via `LEAFLETPDF_TRANSFORM_PROVIDER=mcp-sampling`.
**Reason:** Users with a Claude Desktop or Cursor subscription should not need a separate API key to run Leaflet PDF via MCP. MCP Sampling routes inference requests through the host's existing session, eliminating the cost and configuration barrier. This extends DEC-019's BYOA model to include subscription-based hosts as a valid "provider."
**Supersedes:** Nothing. Extends DEC-019 (adds a fourth provider option; BYOA API key providers unchanged).
**Implemented in:** CR-003.

### DEC-017: User-directed PDF save location
**Decision:** After a successful pipeline run, a native Windows save dialog prompts the user to choose where to save the Reading Artifact. No default output folder.
**Reason:** Users want control over where files land. No default folder avoids cluttering arbitrary locations; each save is an intentional act.
